🔧 Fundamentals

JVM

A runtime environment that executes Java bytecode. It provides memory management, garbage collection, and security features. The JVM is platform-specific but allows Java programs to run on any operating system.

Bytecode

Intermediate code generated by the Java compiler (.class files). It's platform-independent and executed by the JVM. Bytecode is more efficient than interpreted languages but slower than native machine code.

Platform Independence

Java's "write once, run anywhere" principle. Source code compiles to bytecode that runs on any JVM, regardless of the underlying operating system or hardware architecture.

Key Topics

  • Data types, variables, control flow, arrays
  • Packages - Namespace mechanism to organize classes and avoid naming conflicts
  • Access Modifiers - Control visibility of classes, methods, and fields
Summary: Java runs on the JVM using bytecode for cross-platform execution. Master basic syntax including types, variables, control structures, and arrays. Organize code with packages and control visibility using access modifiers.

🏗️ Object-Oriented Programming

Classes

Blueprints or templates that define the structure and behavior of objects. Contain fields (data) and methods (functions). Use PascalCase naming convention.

Objects

Instances of classes created using the new keyword. Each object has its own state (field values) and can invoke methods. Objects are reference types in Java.

Core OOP Principles

  • Encapsulation: Data hiding principle that bundles data and methods together
  • Inheritance: Mechanism where a class inherits properties and methods from another class
  • Polymorphism: Ability of objects to take multiple forms
  • Interfaces: Contracts that define what methods a class must implement
  • Abstract Classes: Classes that cannot be instantiated directly
Summary: Design software using classes and objects with proper constructors. Apply OOP principles: encapsulation (data hiding), inheritance (code reuse), and polymorphism (flexible behavior). Understand when to use interfaces versus abstract classes.

📝 Strings

String Immutability

Strings cannot be changed after creation. Any modification creates a new String object. Ensures thread-safety and prevents accidental modifications.

String Pooling

JVM maintains a pool of unique string literals to save memory. String literals with same content share the same memory location.

StringBuilder vs StringBuffer

  • StringBuilder: Mutable sequence of characters for efficient string building. Not thread-safe but faster
  • StringBuffer: Thread-safe mutable sequence of characters. Synchronized methods ensure thread safety

Common APIs

substring(int start, int end) // Extract portion of string split(String regex) // Split string into array replace(String old, String new) // Replace occurrences format(String format, Object... args) // Format string with placeholders
Summary: Strings are immutable and pooled for memory efficiency. Use StringBuilder for efficient concatenation. Master essential APIs like substring, split, replace, and format.

⚠️ Exceptions and Error Handling

Checked vs Unchecked Exceptions

  • Checked Exceptions: Must be handled at compile time (IOException, SQLException)
  • Unchecked Exceptions: Runtime exceptions that don't need to be declared (NullPointerException, ArrayIndexOutOfBoundsException)

Exception Handling

try { riskyCode(); } catch (Exception e) { handle(e); } finally { cleanup(); }

try-with-resources

try (FileInputStream fis = new FileInputStream("file.txt")) { // use fis } // automatically closed
Summary: Distinguish between checked exceptions (must handle) and unchecked exceptions (runtime errors). Use try/catch/finally and try-with-resources for proper resource management.

🔗 Generics

Type Parameters

List<String> list = new ArrayList<>(); public class Box<T> { private T item; }

Bounds

public class NumberBox<T extends Number> // T must be Number or its subclass

Wildcards

  • ? extends: Upper bounded wildcard for covariance (read-only)
  • ? super: Lower bounded wildcard for contravariance (write access)

Type Erasure

Process where generic type information is removed at compile time. All generic types become Object or their upper bound. Enables backward compatibility but limits runtime type checking.
Summary: Generics provide compile-time type safety through type parameters and bounds. Use wildcards for flexible type relationships. Understand type erasure limitations at runtime.

📚 Collections Framework

Core Interfaces

  • Collection: Root interface for all collections
  • List: Ordered collection allowing duplicates (ArrayList, LinkedList)
  • Set: Collection with no duplicates (HashSet, TreeSet)
  • Map: Key-value pairs, no duplicate keys (HashMap, TreeMap)

Key Implementations

  • ArrayList: Resizable array, O(1) get, O(n) add/remove
  • LinkedList: Doubly-linked list, O(n) get, O(1) add/remove
  • HashMap: Hash table, O(1) average case
  • TreeMap: Red-black tree, O(log n) operations, sorted

equals() and hashCode() Contracts

equals() Contract: Reflexive, symmetric, transitive, consistent, non-null
hashCode() Contract: Equal objects must have same hashCode, consistent across calls
Summary: Use Collection, List, Set, and Map interfaces with appropriate implementations. Choose based on performance characteristics and ordering needs. Always implement equals/hashCode correctly.

📅 Date and Time (java.time)

Core Classes

  • LocalDate: Date without time or timezone
  • LocalTime: Time without date or timezone
  • LocalDateTime: Combines LocalDate and LocalTime
  • Instant: Point in time on UTC timeline
  • ZoneId: Timezone identifier
LocalDate today = LocalDate.now(); LocalTime meeting = LocalTime.of(14, 30); LocalDateTime event = LocalDateTime.of(2024, 12, 25, 10, 30); Instant now = Instant.now(); ZoneId zone = ZoneId.of("America/New_York");
Summary: Use immutable java.time types instead of legacy Date/Calendar. Handle time zones properly with ZoneId. Format and parse dates with DateTimeFormatter.

📁 I/O and NIO.2

java.io Streams

  • InputStream/OutputStream: Byte-oriented streams for binary data
  • Reader/Writer: Character-oriented streams for text data

java.nio.file

Path path = Paths.get("folder", "file.txt"); Path absolute = path.toAbsolutePath(); Files.copy(source, destination); List<String> lines = Files.readAllLines(path);
Summary: Use java.io for basic I/O operations with proper resource management. Prefer java.nio.file for modern file operations. Always use try-with-resources for automatic cleanup.

🧵 Concurrency

Threads and Executors

Thread thread = new Thread(() -> System.out.println("Hello")); thread.start(); ExecutorService executor = Executors.newFixedThreadPool(4); Future<String> future = executor.submit(() -> "Result");

Synchronization

  • synchronized: Control access to shared resources
  • volatile: Ensure variable visibility across threads
  • atomics: Thread-safe classes for atomic operations

CompletableFuture

CompletableFuture.supplyAsync(() -> "Hello") .thenApply(s -> s + " World") .thenAccept(System.out::println);
Summary: Create and manage threads using Executors. Ensure thread-safety with synchronization, volatile, and atomic classes. Compose asynchronous operations with CompletableFuture.

⚙️ JVM Internals and Performance

Memory Model

  • Heap: Runtime data area for objects and arrays
  • Stack: Thread-specific memory for method calls
  • Metaspace: Memory area for class metadata (Java 8+)

Garbage Collection

Automatic memory management that reclaims unused objects. Algorithms: Serial, Parallel, CMS, G1, ZGC, Shenandoah. Young GC (frequent, fast) vs Full GC (infrequent, slow).

Profiling Tools

  • JFR (Java Flight Recorder): Low-overhead profiling built into JVM
  • VisualVM: GUI tool for monitoring and profiling
  • jcmd, jmap, jstack: Command-line tools for diagnostics
Summary: Understand JVM memory organization and garbage collection algorithms. Profile applications using JFR, VisualVM, or async-profiler. Monitor performance with jcmd, jmap, jstack.

💡 Coding Practices

Immutability

public final class Person { private final String name; private final int age; }

equals/hashCode/compareTo

public boolean equals(Object obj) { if (this == obj) return true; if (!(obj instanceof Person)) return false; Person other = (Person) obj; return Objects.equals(name, other.name); }

API Design

public Optional findUserById(Long id) throws IllegalArgumentException { if (id == null) throw new IllegalArgumentException("ID cannot be null"); return userRepository.findById(id); }
Summary: Favor immutable objects and defensive copying. Implement equals/hashCode/compareTo correctly. Design clear, consistent APIs with proper error handling.

✨ Language Features by Version

Modern Features

  • var Java 10: Local variable type inference
  • switch expressions Java 14: Expression-based switch
  • text blocks Java 15: Multi-line strings
  • records Java 16: Data classes with automatic equals/hashCode/toString
  • sealed classes Java 17: Restricted inheritance
  • pattern matching Java 17+: Enhanced instanceof with pattern matching
var list = new ArrayList(); String result = switch (day) { case MONDAY -> "Start of week"; default -> "Other day"; }; public record Person(String name, int age) {} public sealed class Shape permits Circle, Rectangle {} if (obj instanceof String s) { System.out.println(s.toUpperCase()); }
Summary: Adopt modern Java features like var, switch expressions, and text blocks. Use records for data classes and sealed classes for controlled inheritance. Apply pattern matching where appropriate.

🎨 Design Patterns

Creational Patterns

  • Factory: Create objects without specifying exact classes
  • Builder: Construct complex objects step by step

Structural Patterns

  • Adapter: Make incompatible interfaces work together
  • Decorator: Add behavior to objects dynamically

Behavioral Patterns

  • Strategy: Define family of algorithms and make them interchangeable
  • Observer: Define one-to-many dependency between objects
Summary: Apply design patterns to solve common problems. Prefer composition over inheritance. Use concurrency patterns safely.
